# X記事ページをMarkdownとして出力するChrome拡張 要件・仕様・設計

## 1. 目的

X（旧Twitter）の記事ページ（単体ポストおよびスレッド）を、再利用しやすいMarkdown形式でエクスポートするChrome拡張を提供する。  
可能な限り本文だけでなく、添付画像もセットで保存し、記事アーカイブやメモ作成を効率化する。

対象例:
- `https://x.com/<user>/status/<tweet_id>`
- 例: `https://x.com/LunarResearcher/status/2043690015675318360`

---

## 2. スコープ

### 2.1 対象（In Scope）
- 現在表示中のX記事ページから以下を抽出
  - 投稿本文テキスト
  - 投稿メタ情報（投稿者名、ユーザーID、投稿日時、URL）
  - 添付画像URL（可能な範囲）
  - スレッド内の連続ポスト（任意機能。設定でON/OFF）
- Markdown文字列の生成
- ユーザーによる保存手段
  - クリップボードコピー
  - `.md` ファイル保存
  - 画像を個別ダウンロード（複数ファイル）

### 2.2 非対象（Out of Scope）
- X API（有償含む）を使ったサーバーサイド収集
- 動画ファイルのダウンロード（初期版）
- 認証情報の保存・管理
- 一括URLバッチ処理（初期版）

---

## 3. ユースケース

1. ユーザーがX記事ページを開く
2. 拡張アイコンをクリック
3. 「Markdownとして出力」を実行
4. 拡張が本文・メタ情報・画像参照を抽出
5. Markdownプレビューを表示
6. ユーザーが以下のいずれかを選択
   - クリップボードへコピー
   - `.md` 保存
   - 画像ダウンロード（必要時）

---

## 4. 機能要件

### 4.1 必須要件（MVP）
- **F-01**: `x.com/*/status/*` ページでのみ拡張アクションを有効化
- **F-02**: 現在表示中のメイン投稿の本文テキストを抽出
- **F-03**: 投稿メタ情報を抽出
  - 投稿者表示名
  - 投稿者ハンドル（@username）
  - 投稿日時（ISO 8601）
  - 元URL
- **F-04**: 添付画像URL一覧を抽出
- **F-05**: 下記フォーマットでMarkdownを生成
- **F-06**: Markdownをクリップボードへコピー
- **F-07**: Markdownを `.md` ファイルとして保存

### 4.2 推奨要件（v1.x）
- **F-08**: スレッド抽出（親/連続投稿を時系列で結合）
- **F-09**: 画像のローカル保存（ファイル名正規化）
- **F-10**: 画像参照形式の切り替え
  - リモートURL参照
  - ローカル相対パス参照
- **F-11**: Markdownテンプレートの選択
  - シンプル
  - Front Matter付き

---

## 5. 非機能要件

- **N-01 性能**: 単体投稿の抽出は2秒以内を目標
- **N-02 堅牢性**: DOM構造変更時でも失敗理由をUI表示（例: `抽出対象が見つかりません`）
- **N-03 保守性**: 抽出ロジックをセレクタ定義と変換処理で分離
- **N-04 安全性**: 不要な権限を要求しない（最小権限）
- **N-05 可観測性**: 開発モードでデバッグログを有効化

---

## 6. 制約・前提

- XのDOMは頻繁に変化するため、セレクタ依存ロジックは壊れやすい
- CORSや認証状態の影響で画像の直接取得に失敗する可能性がある
- Chrome Web StoreポリシーおよびX利用規約に準拠すること
- まずは「ユーザーが自身のブラウザで閲覧中のページ」からの抽出に限定する

---

## 7. 出力仕様（Markdown）

### 7.1 推奨フォーマット（本文）

```md
# X Post Export

- Author: Lunar Researcher (@LunarResearcher)
- Date: 2026-04-10T12:34:56.000Z
- URL: https://x.com/LunarResearcher/status/2043690015675318360

## Content

（投稿本文）

## Images

![image-01](images/2043690015675318360-01.jpg)
![image-02](images/2043690015675318360-02.jpg)
```

### 7.2 ファイル命名規則

- Markdown: `<username>-<tweet_id>.md`
- 画像: `images/<tweet_id>-<index>.<ext>`（`<index>` は2桁0埋め: `01`, `02`, ...）
- 画像の拡張子はXのCDN URL（`pbs.twimg.com`）のクエリパラメータ（`?format=jpg` 等）から判定する。判定できない場合は `.jpg` をデフォルトとする

## 9. 技術設計（Chrome Extension Manifest V3）

### 9.1 構成

- `manifest.json`
- `src/content/extractor.ts`
  - DOMから投稿データ抽出
- `src/background/service-worker.ts`
  - ダウンロード処理
  - メッセージルーティング
  - `.md` ファイルおよび画像の保存：Blob → `URL.createObjectURL()` → `chrome.downloads.download()` の経路はサービスワーカー側で行う（ポップアップ側で生成したBlob URLはポップアップが閉じると無効になるため）
- `src/popup/popup.ts`
  - UI制御
  - 生成結果表示
- `src/core/markdown.ts`
  - Markdown組み立て
- `src/core/image.ts`
  - 画像URL正規化、拡張子判定、命名
- `src/types/index.ts`
  - データ型定義

### 9.2 データモデル（例）

```ts
type XPost = {
  id: string;
  url: string;
  authorName: string;
  authorHandle: string;
  createdAt: string;
  text: string;
  images: Array<{
    url: string;
    alt?: string;
  }>;
};
```

### 9.3 抽出戦略

1. URLが `status/<id>` を含むことを確認
2. メイン投稿コンテナを探索（`article`ベース + 属性ヒューリスティクス）
3. テキストノードを収集して整形
4. 画像要素を収集してURLを正規化
5. 投稿日時・投稿者情報を抽出
6. データモデルへマッピングしMarkdown変換

DOM変化に備え、セレクタは複数候補を持つフォールバック方式とする。

**セレクタ戦略**  
XはReact製SPAであり、クラス名は頻繁に変化する。代わりに `data-testid` 属性が比較的安定したセレクタとして利用できる。実装では `data-testid` を第一候補とし、クラスベースのセレクタをフォールバックとして持つ。

| 対象 | 推奨セレクタ（第一候補） |
|---|---|
| 投稿コンテナ | `article[data-testid="tweet"]` |
| 本文テキスト | `[data-testid="tweetText"]` |
| 投稿者名 | `[data-testid="User-Name"]` |

**SPAナビゲーション対応**  
Xはページ遷移時にフルリロードを行わないため、コンテンツスクリプトは一度注入されると同一セッション内で使い回される。投稿間の遷移（`history.pushState` / `popstate`）を検知し、ポップアップからのメッセージ受信時に最新のDOMを参照する設計とする。コンテンツスクリプト側でURLキャッシュを持ち、URLが変化した場合は抽出結果をクリアする。

## 10. 権限設計（最小権限）

想定 `manifest.json`（概略）:
- `permissions`
  - `activeTab`
  - `scripting`
  - `downloads`（画像/Markdown保存に必要）
- `host_permissions`
  - `https://x.com/*`

**`clipboardWrite` について**  
クリップボードコピー（F-06）はポップアップのコンテキスト（`popup.ts`）で `navigator.clipboard.writeText()` を呼ぶことで実現する。この場合 `clipboardWrite` パーミッションは不要。コンテンツスクリプトのコンテキストで呼ぶ実装にした場合は `permissions` への追加が必要になるため、コピー処理はポップアップ側に置くこととする。

---

## 11. エラーハンドリング方針

- 対象外URL: `このページはXの投稿ページではありません`
- 抽出失敗: `投稿本文を取得できませんでした。ページを再読み込みしてください`
- 画像保存失敗: 失敗画像のみ一覧表示し、Markdown出力は継続
- 例外発生時: ユーザー向け簡易メッセージ + 開発ログ詳細

---

## 12. テスト計画

### 12.1 手動テスト
- 単体投稿（画像なし）
- 単体投稿（画像1枚/複数枚）
- 長文投稿
- 日本語・英語・絵文字混在
- ログイン状態 / 未ログイン状態
- スレッド投稿（ON/OFF）

### 12.2 観点
- 本文欠落がないか
- 改行・リンク・メンションの整形が崩れないか
- 画像リンク切れがないか
- 保存ファイル名が安全か（禁止文字除去）

---

## 13. リリース段階案

1. **Phase 1 (MVP)**  
   単体投稿の本文+メタ情報+画像URLをMarkdown化して保存
2. **Phase 2**  
   ローカル画像保存 + 相対パス埋め込み
3. **Phase 3**  
   スレッド結合、テンプレート機能、失敗時リトライ強化

## 15. 受け入れ基準（Definition of Done）

- X投稿URLで拡張を起動し、Markdownを出力できる
- Markdownに本文・投稿者・日時・URLが含まれる
- 画像付き投稿で画像参照セクションが生成される
- `.md` 保存が成功し、主要ケースで再現性がある
- 失敗時にユーザーが理解可能なエラーメッセージを表示できる
